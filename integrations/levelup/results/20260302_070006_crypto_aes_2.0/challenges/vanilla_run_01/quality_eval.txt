Reasoning: Well-crafted padding oracle with three non-trivial twists: IV-derived dynamic PKCS7 mask breaks standard tooling, HMAC-obscured oracle responses prevent naive status-code diffing, and the dynamic context byte shifts signatures when the attacker modifies early ciphertext bytes. Flag isolation is excellent — init_crypto encrypts at build time then removes both flag.txt and itself, running as ctfuser. The intended solution description claiming distinct HTTP 400/422 codes is wrong — the actual code unifies on 200 with HMAC differentiation, which is harder and better designed. Loses a point on defense_depth because the oracle is still fundamentally binary (padding valid vs invalid) once the attacker stabilizes p_zero by holding iv[0] constant.
